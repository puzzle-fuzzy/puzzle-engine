项目概览

Excuse 是一个全栈 AI 内容生成平台，支持文本、图片、视频的一站式生成，以及创新的 Canvas 故事转视频流水线。

🎯 核心功能

  • 📝 文本生成 — 千问系列模型 (Max/Plus/Turbo/Long/3.7)
  • 🎨 图片生成 — 千问图像 2.0 Pro / Max
  • 🎬 视频生成 — HappyHorse + 万相 2.7 (文/图/参考生视频)
  • 🎞️ 视频编辑 — 风格变换、局部替换
  • 🖼️ Canvas 流水线 — 故事 → 分析 → 角色 → 场景 → 分镜 → 视频
  • 💰 计费系统 — Token/图片/视频秒多维度计费

🏗️ 设计亮点

  • 🔗 端到端类型安全 — Eden Treaty 零手写 fetch
  • 📋 声明式模型配置 — 所有 AI 模型统一声明，零分支
  • ⚡ Fire-and-Forget — 异步执行 + SSE 实时推送
  • 🔌 PG LISTEN/NOTIFY — 数据库驱动的实时通信
  • 🧪 依赖注入测试 — Worker deps 可替换
  • 📊 Repository 模式 — DB 访问通过函数导出

💡 一句话理解

用户输入一段故事文本 → 平台自动分析角色和场景 → 生成角色参考图和场景参考图 → 拆分分镜脚本 → 批量生成视频片段。整个流程通过 SSE 实时推送进度，Worker 后台轮询 DashScope 异步任务。

🔄 执行模式：Fire-and-Forget

1

客户端发起请求

POST /api/canvas/projects/:id/analyze — 触发某个 Pipeline 阶段

2

Server 立即返回

返回 { accepted: true, runId: "xxx" } — 不等待执行完成

3

后台异步执行

fireAndForgetWithRun() 在后台调用 LLM / AI 模型，等待结果

4

SSE 推送结果

完成后通过 dispatchToUser() 推送 pipeline_node_update 事件

📝 LLM Prompt 设计

每个阶段使用精心设计的中文 Prompt，要求 LLM 输出结构化 JSON：

  • 分析阶段 — 提取摘要、冲突、时间线、角色名、场景名
  • 角色阶段 — 脸型/五官/发型/服装 + 英文 identityPrompt (≤100词)
  • 场景阶段 — 建筑/色彩/光线 + cameraRules + 英文 scenePrompt
  • 分镜阶段 — 专业导演视角，逐秒动作分解，3-15秒/镜头

🛡️ 并发守卫

每个阶段执行前通过 findActiveRunForPhase() 检查是否有进行中的 Run：

  • 如果存在活跃 Run → 返回 409 Conflict
  • 否则创建新 Pipeline Run 并执行
  • 归属校验：所有操作先确认资源属于当前用户
  • 自动完成：Worker 检查所有 Shot 完成后标记项目 completed

异步任务流

Worker 持续轮询 PostgreSQL 中的 pending 视频任务，查询 DashScope 状态并处理结果

🤖 Worker 主循环

1

轮询 DB

pollPendingVideoTasks() — 查询所有 status=pending/processing 的视频生成记录

2

查询 DashScope

client.queryTask(taskId) — 向 DashScope API 查询异步任务状态

3a

✅ SUCCEEDED

下载视频 → 上传到 OSS/本地 → 计算费用 → 更新 DB → PG NOTIFY → 更新 Shot 状态 → 检查项目完成

3b

❌ FAILED

标记失败 → PG NOTIFY → 更新 Shot 状态为 failed → 检查项目完成

3c

⏳ PENDING / RUNNING

标记为 processing，跳过等待下一轮轮询。超时检测：超过 4 小时自动标记失败

⏱️ 轮询策略

每 5 秒轮询一次（可配置），分段 sleep 1 秒以快速响应退出信号。

优雅退出：收到 SIGINT/SIGTERM → 等待当前任务完成（最长 30s）→ 退出

🏥 健康检查

Worker 在 :5100 暴露健康端点：

  • isPolling — 是否正在轮询
  • lastPollAt — 上次轮询时间
  • totalTasksProcessed — 已处理任务数
  • lastPollError — 最近错误

📬 事件类型

事件名触发场景
generation_status通用生成任务状态变更（成功/失败/处理中）
pipeline_node_updateCanvas 流水线进度（阶段完成/Shot 完成）

🔧 连接管理

  • 按 userId 维护 Map<string, Set<Sender>>
  • 支持多标签页（一个用户多个 SSE 连接）
  • 30 秒心跳保活
  • 客户端自动重连（EventSource + 自定义逻辑）
  • JWT Token 鉴权：/api/sse?token=<jwt>

🌉 为什么用 PG LISTEN/NOTIFY 而非 WebSocket？

Server 和 Worker 是独立进程，需要跨进程通信。PostgreSQL 的 LISTEN/NOTIFY 天然支持这种"发布-订阅"模式，无需引入 Redis 等额外中间件。Worker 完成任务后执行 NOTIFY，Server 的 LISTEN 回调收到消息后分发给对应的 SSE 连接。

🔧 声明式 InputMapping 机制

每个参数通过 InputMapping 联合类型声明目标位置：prompt | parameter | media | mediaField | ignored。DashScopeClient 的 applyMappings() 自动路由，buildRequestBody() 按 requestType 塑形。零模型分支代码。
