# 创意流水线架构设计

> 版本：v0.1 · 日期：2026-06-11 · 状态：草案

## 1. 背景与动机

### 1.1 现状

当前系统架构是「单步单模型」设计：一个请求选择一个模型，执行一次 API 调用，产生一条 `generation_record`。这套架构已经覆盖了基础的文本、图像、视频生成需求。

**现有请求流：**

```
用户选模型 + 填参数
  → POST /api/generate { model, parameters }
    → DashScopeClient.generate(model, params)
      → 一次 API 调用
        → 写入一条 generation_record
          → 返回结果
```

### 1.2 新需求

需要支持一系列创意性功能，它们的共同特征是**多步骤、有依赖关系**：

| 功能 | 步骤拆解 |
|------|----------|
| **小说生视频** | 小说 → LLM 拆分场景 → LLM 生成分镜提示词 → 逐场景生成图片/视频 → 聚合 |
| **小说生剧本** | 小说 → LLM 生成剧本大纲 → LLM 生成逐幕剧本 → （可选）生成分镜表 |
| **自由发挥视频** | LLM 构思创意 → LLM 生成镜头脚本 → 逐镜头生成视频 → 聚合 |
| **图片生视频** | 已支持（i2v 模型），但后续可扩展为「图片 + 音乐 → 带配乐视频」 |

**核心差异：步骤之间有数据依赖（上一步的输出是下一步的输入），需要状态机管理而非单次调用。**

### 1.3 设计原则

1. **不重构现有架构**：现有单步生成流程（选模型 → 填参数 → 生成）完全保留不动
2. **增量叠加**：在现有架构之上增加一层流水线编排能力
3. **复用现有组件**：每一步底层仍走 `ModelConfig → inputMapping → DashScopeClient`，不引入新的 API 调用方式
4. **渐进式实现**：先实现核心流水线引擎 + 一个创意功能，再逐步扩展

## 2. 功能定义

### 2.1 小说生视频（novel-to-video）

**用户输入：** 一段小说文本（粘贴或上传文件）

**流水线步骤：**

```
Step 1: scene_analysis（场景分析）
  模型：qwen-max
  输入：小说全文
  输出：N 个场景描述，每个场景包含：
    - sceneIndex: number
    - title: string              // 场景标题
    - description: string        // 场景画面描述
    - prompt: string             // 生成用的提示词（英文）
    - characters: string[]       // 出场角色
    - duration: number           // 建议视频时长（秒）

Step 2: scene_videos（逐场景生成视频）—— 并行 N 个子任务
  模型：wan2.7-t2v 或 happyhorse-1.0-t2v（用户可选）
  输入：每个场景的 prompt + duration
  输出：N 个视频 URL
  特点：并行执行，所有视频同时开始生成

Step 3: aggregate（聚合）
  无 API 调用，纯数据聚合
  输入：所有场景视频 URL + 场景元数据
  输出：完整的场景列表，包含视频 URL、播放顺序
```

**可选扩展：**
- 增加音频生成步骤（为每个场景生成背景音乐）
- 增加字幕生成步骤（LLM 根据场景描述生成字幕）
- 增加视频拼接步骤（调用 FFmpeg 或第三方 API 拼接所有片段）

### 2.2 小说生剧本（novel-to-script）

**用户输入：** 小说文本 + 剧本风格（电影/电视剧/短片）

**流水线步骤：**

```
Step 1: script_outline（剧本大纲）
  模型：qwen-max
  输入：小说全文 + 风格偏好
  输出：
    - title: string
    - genre: string
    - acts: Act[]（幕结构）
      - actIndex, title, synopsis, scenes: Scene[]

Step 2: script_detail（逐幕详细剧本）—— 可并行
  模型：qwen-max
  输入：大纲中的单幕内容
  输出：
    - 对白、动作描述、场景指示
    - 角色情绪标注

Step 3: script_assemble（剧本组装）
  无 API 调用
  输入：所有幕的详细剧本
  输出：完整剧本文档（Markdown / PDF）
```

### 2.3 自由发挥生成视频（free-create-video）

**用户输入：** 创意关键词 / 一句话描述 / 或完全随机

**流水线步骤：**

```
Step 1: creative_brainstorm（创意构思）
  模型：qwen-max
  输入：用户关键词（可为空，此时完全自由发挥）
  输出：创意概念 + 完整故事梗概 + 镜头分解

Step 2: scene_videos（逐镜头生成视频）—— 并行
  模型：wan2.7-t2v
  输入：每个镜头的提示词
  输出：视频 URL 列表

Step 3: aggregate（聚合）
  输出：创意说明 + 视频列表
```

## 3. 数据模型设计

### 3.1 新增 `workflows` 表

```sql
CREATE TABLE workflows (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id    UUID NOT NULL REFERENCES accounts(id),
  type          VARCHAR(50) NOT NULL,          -- 'novel-to-video' / 'novel-to-script' / 'free-create-video' / ...
  status        VARCHAR(20) NOT NULL DEFAULT 'running',  -- running / paused / succeeded / failed
  title         VARCHAR(255),                  -- 用户可编辑的标题
  input_params  JSONB NOT NULL,                -- 原始用户输入（小说全文、创意关键词等）
  output_result JSONB,                         -- 最终聚合结果
  config        JSONB,                         -- 流水线配置（用户选择的模型、分辨率偏好等）
  error_message TEXT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_workflows_account ON workflows(account_id);
CREATE INDEX idx_workflows_status ON workflows(status);
```

### 3.2 扩展 `generation_records` 表

在现有表上新增两个 nullable 字段，**向后兼容**：

```sql
ALTER TABLE generation_records
  ADD COLUMN workflow_id UUID REFERENCES workflows(id),
  ADD COLUMN step_name   VARCHAR(100);

CREATE INDEX idx_generation_records_workflow ON generation_records(workflow_id);
```

- `workflow_id`：该记录属于哪个 workflow，为 NULL 表示独立生成（现有逻辑）
- `step_name`：在流水线中的步骤标识，如 `scene_analysis`、`scene_video_3`

### 3.3 关系图

```
workflows (1) ──< generation_records (N)
  │                    │
  │                    ├── workflow_id = NULL  → 独立生成（现有逻辑，不受影响）
  │                    └── workflow_id != NULL → 流水线中的步骤
  │
  ├── input_params:  { "novelText": "...", "style": "cinematic" }
  ├── config:        { "videoModel": "wan2.7-t2v", "resolution": "1080P" }
  └── output_result: { "scenes": [...], "videos": [...] }
```

## 4. 流水线引擎设计

### 4.1 核心抽象

```typescript
// packages/pipeline/src/types.ts

/** 流水线定义 */
interface PipelineDefinition {
  type: string                          // 'novel-to-video'
  name: string                          // '小说生视频'
  description: string
  steps: PipelineStepDefinition[]
  /** 从用户输入构造初始 context */
  createInitialContext: (inputParams: Record<string, unknown>) => PipelineContext
}

/** 单步骤定义 */
interface PipelineStepDefinition {
  name: string                          // 'scene_analysis'
  label: string                         // '场景分析'
  /** 使用的模型 ID（对应 model-configs.ts 中的 id） */
  model: string
  /** 是否为异步任务（视频 = true） */
  async: boolean
  /**
   * 从 pipeline context 中提取当前步骤的参数。
   * 返回值就是普通 parameters，走现有 DashScopeClient.generate() 流程。
   */
  resolveParams: (context: PipelineContext) => Record<string, unknown>
  /**
   * 当前步骤成功后的处理。
   * 返回下一步动作。
   */
  onSucceeded: (output: StepOutput, context: PipelineContext) => StepAction
  /** 当前步骤失败后的处理（可选，默认标记整个 workflow 失败） */
  onFailed?: (error: string, context: PipelineContext) => StepAction
}

/** 流水线运行时上下文 */
interface PipelineContext {
  workflowId: string
  accountId: string
  inputParams: Record<string, unknown>  // 用户原始输入
  config: Record<string, unknown>       // 用户配置
  stepOutputs: Map<string, StepOutput>  // 已完成步骤的输出（按 stepName 索引）
  metadata: Record<string, unknown>     // 引擎内部状态
}

/** 步骤输出 */
interface StepOutput {
  stepName: string
  status: 'succeeded' | 'failed'
  data: Record<string, unknown>         // outputResult
  cost?: CostDetail
  recordId: string                      // 对应的 generation_record ID
}

/** 步骤完成后的动作指令 */
type StepAction =
  | { type: 'next'; stepIndex: number }
  | { type: 'parallel'; tasks: ParallelTask[] }
  | { type: 'wait-parallel'; stepName: string }  // 等待指定步骤名的所有并行任务完成
  | { type: 'aggregate' }                         // 聚合并完成
  | { type: 'done'; output: Record<string, unknown> }
  | { type: 'fail'; message: string }

/** 并行子任务 */
interface ParallelTask {
  stepName: string           // 'scene_video_0', 'scene_video_1', ...
  model: string
  params: Record<string, unknown>
  metadata?: Record<string, unknown>
}
```

### 4.2 引擎状态机

```
                    ┌──────────┐
                    │ created  │
                    └────┬─────┘
                         │ 执行 step[0]
                    ┌────▼─────┐
              ┌─────│ running  │─────┐
              │     └────┬─────┘     │
         步骤成功          │          步骤失败
              │     ┌────▼─────┐     │
              │     │  检查     │     │
              │     │ StepAction│     │
              │     └────┬─────┘     │
              │          │           │
       ┌──────┼──────────┤           │
       │      │          │           │
  ┌────▼──┐ ┌▼───────┐ ┌▼──────┐  ┌▼──────┐
  │ next  │ │parallel │ │ done  │  │ fail  │
  │ 执行  │ │ 并行N个 │ │ 完成  │  │ 失败  │
  │ 下一步 │ │ 子任务  │ │       │  │       │
  └────┬──┘ └────┬────┘ └───────┘  └───────┘
       │         │
       │    等待全部完成
       │         │
       │    ┌────▼────┐
       └───►│ running │◄─── 循环直到 done/fail
            └─────────┘
```

### 4.3 引擎核心逻辑（伪代码）

```typescript
// packages/pipeline/src/engine.ts

export class PipelineEngine {
  constructor(
    private definition: PipelineDefinition,
    private deps: PipelineEngineDeps,   // DB + provider 注入
  ) {}

  /**
   * 启动一条流水线
   */
  async start(inputParams: Record<string, unknown>, config: Record<string, unknown>) {
    const context = this.definition.createInitialContext(inputParams)
    context.config = config

    // 写入 workflow 记录
    const workflow = await this.deps.createWorkflow({
      type: this.definition.type,
      inputParams,
      config,
      status: 'running',
    })
    context.workflowId = workflow.id

    // 从第一步开始执行
    await this.executeStep(0, context)
  }

  /**
   * 执行指定步骤
   */
  private async executeStep(stepIndex: number, context: PipelineContext) {
    const stepDef = this.definition.steps[stepIndex]

    // 1. 从 context 中解析参数
    const params = stepDef.resolveParams(context)

    // 2. 创建 generation_record（关联 workflow）
    const record = await this.deps.createGenerationRecord({
      workflowId: context.workflowId,
      stepName: stepDef.name,
      model: stepDef.model,
      parameters: params,
      status: 'pending',
    })

    // 3. 调用现有的 DashScopeClient.generate()
    try {
      const result = await this.deps.providerGenerate(stepDef.model, params)

      if (result.providerTaskId) {
        // 异步任务（视频）：标记 processing，等待 worker 回调
        await this.deps.markProcessing(record.id, { taskId: result.providerTaskId })
        // 引擎在此挂起，worker 完成后通过 resume 方法继续
        return
      }

      // 同步任务：立即处理结果
      await this.handleStepSucceeded(stepDef, record.id, result.output, context)
    }
    catch (error) {
      await this.handleStepFailed(stepDef, record.id, String(error), context)
    }
  }

  /**
   * 步骤成功回调（同步任务直接调用 / 异步任务由 worker 触发）
   */
  async handleStepSucceeded(
    stepDef: PipelineStepDefinition,
    recordId: string,
    output: Record<string, unknown>,
    context: PipelineContext,
  ) {
    const stepOutput: StepOutput = {
      stepName: stepDef.name,
      status: 'succeeded',
      data: output,
      recordId,
    }
    context.stepOutputs.set(stepDef.name, stepOutput)

    // 更新 generation_record
    await this.deps.markSucceeded(recordId, output)

    // 查询动作
    const action = stepDef.onSucceeded(stepOutput, context)
    await this.executeAction(action, context)
  }

  /**
   * 执行动作指令
   */
  private async executeAction(action: StepAction, context: PipelineContext) {
    switch (action.type) {
      case 'next':
        // 执行下一步
        await this.executeStep(action.stepIndex, context)
        break

      case 'parallel': {
        // 并行发起多个子任务
        for (const task of action.tasks) {
          await this.deps.createGenerationRecord({
            workflowId: context.workflowId,
            stepName: task.stepName,
            model: task.model,
            parameters: task.params,
            status: 'pending',
          })
          // 不 await，直接并行发起
          this.deps.providerGenerate(task.model, task.params)
            .then(result => { /* 同步/异步处理 */ })
            .catch(error => { /* 失败处理 */ })
        }
        // 引擎挂起，等待所有子任务完成
        break
      }

      case 'wait-parallel':
        // 等待指定 stepName 的所有并行任务完成
        // 由 worker 的并行任务完成事件触发恢复
        break

      case 'done':
        // 流水线完成
        await this.deps.updateWorkflow(context.workflowId, {
          status: 'succeeded',
          outputResult: action.output,
        })
        await this.deps.notifyWorkflowStatus(context)
        break

      case 'fail':
        await this.deps.updateWorkflow(context.workflowId, {
          status: 'failed',
          errorMessage: action.message,
        })
        await this.deps.notifyWorkflowStatus(context)
        break
    }
  }

  /**
   * Worker 回调：异步任务完成后恢复流水线
   */
  async resumeFromAsyncCompletion(recordId: string, output: Record<string, unknown>) {
    // 1. 查出 record → 找到 workflow_id + step_name
    const record = await this.deps.getRecord(recordId)
    const workflow = await this.deps.getWorkflow(record.workflowId)
    const context = this.reconstructContext(workflow)

    // 2. 重建 context（从 workflow 的所有子 record 中恢复 stepOutputs）
    const childRecords = await this.deps.getWorkflowRecords(record.workflowId)
    for (const r of childRecords) {
      if (r.status === 'succeeded' && r.outputResult) {
        context.stepOutputs.set(r.stepName, {
          stepName: r.stepName,
          status: 'succeeded',
          data: r.outputResult,
          recordId: r.id,
        })
      }
    }

    // 3. 检查是否所有并行任务都完成
    const stepDef = this.findStepByName(record.stepName)
    const action = stepDef.onSucceeded(
      { stepName: record.stepName, status: 'succeeded', data: output, recordId: record.id },
      context,
    )
    await this.executeAction(action, context)
  }
}
```

### 4.4 具体流水线示例：小说生视频

```typescript
// packages/pipeline/src/pipelines/novel-to-video.ts

import type { PipelineDefinition, PipelineStepDefinition } from '../types'

export const novelToVideoPipeline: PipelineDefinition = {
  type: 'novel-to-video',
  name: '小说生视频',
  description: '将小说文本拆分为场景，为每个场景生成视频',

  createInitialContext: (inputParams) => ({
    workflowId: '',
    accountId: '',
    inputParams,
    config: {},
    stepOutputs: new Map(),
    metadata: {},
  }),

  steps: [
    // ── Step 1: 场景分析 ──────────────────────────────────
    {
      name: 'scene_analysis',
      label: '场景分析',
      model: 'qwen-max',
      async: false,

      resolveParams: (ctx) => ({
        prompt: `你是一位专业的影视导演。请分析以下小说文本，将其拆分为适合拍摄的视频场景。

要求：
1. 每个场景包含完整的画面描述
2. 为每个场景生成英文提示词（用于 AI 视频生成）
3. 指定建议的视频时长
4. 标注出场角色

输出 JSON 格式：
{
  "title": "视频标题",
  "totalScenes": N,
  "scenes": [
    {
      "sceneIndex": 0,
      "title": "场景标题",
      "description": "中文画面描述",
      "prompt": "English prompt for video generation, cinematic, detailed...",
      "characters": ["角色1", "角色2"],
      "duration": 5
    }
  ]
}

小说文本：
${ctx.inputParams.novelText}`,
        max_tokens: 4000,
        temperature: 0.7,
      }),

      onSucceeded: (output, ctx) => {
        const scenes = JSON.parse(output.data.text).scenes
        ctx.metadata.scenes = scenes

        // 为每个场景创建并行视频生成任务
        const tasks = scenes.map((scene: any, i: number) => ({
          stepName: `scene_video_${i}`,
          model: ctx.config.videoModel || 'wan2.7-t2v',
          params: {
            prompt: scene.prompt,
            duration: scene.duration,
            resolution: ctx.config.resolution || '1080P',
          },
          metadata: { sceneIndex: i },
        }))

        return { type: 'parallel', tasks }
      },
    },

    // ── Step 2: 逐场景视频生成（并行模板，由引擎动态创建实例）──
    {
      name: 'scene_video',        // 模板名，实际实例为 scene_video_0, scene_video_1, ...
      label: '场景视频生成',
      model: 'wan2.7-t2v',
      async: true,

      resolveParams: (ctx) => {
        // 由引擎从 ParallelTask.params 传入，不需要从 context 解析
        return ctx.metadata._parallelParams || {}
      },

      onSucceeded: (output, ctx) => {
        // 检查所有并行任务是否完成
        const scenes = ctx.metadata.scenes as any[]
        const allDone = scenes.every((_, i) =>
          ctx.stepOutputs.has(`scene_video_${i}`),
        )
        if (!allDone) {
          return { type: 'wait-parallel', stepName: 'scene_video' }
        }

        // 全部完成 → 聚合
        const videos = scenes.map((scene: any, i: number) => {
          const stepOutput = ctx.stepOutputs.get(`scene_video_${i}`)
          return {
            ...scene,
            videoUrl: stepOutput?.data.savedUrls?.[0],
          }
        })

        return {
          type: 'done',
          output: {
            title: ctx.metadata.title,
            totalScenes: scenes.length,
            scenes: videos,
          },
        }
      },
    },
  ],
}
```

## 5. 各层改动详细设计

### 5.1 新增 `packages/pipeline`

```
packages/pipeline/
├── package.json
├── src/
│   ├── index.ts              // 导出公共 API
│   ├── types.ts              // 类型定义（PipelineDefinition, StepAction 等）
│   ├── engine.ts             // PipelineEngine 核心引擎
│   ├── registry.ts           // 流水线注册表（按 type 查找 PipelineDefinition）
│   └── pipelines/            // 具体流水线实现
│       ├── novel-to-video.ts
│       ├── novel-to-script.ts
│       └── free-create-video.ts
```

**依赖关系：**
```
@excuse/pipeline  →  @excuse/shared (types)
                  →  @excuse/provider (通过依赖注入，不直接 import)
                  →  @excuse/db (通过依赖注入，不直接 import)
                  →  @excuse/billing (通过依赖注入，不直接 import)
```

引擎不直接调用 provider/db/billing，而是通过 `PipelineEngineDeps` 接口注入。这保持了包的独立性，也方便测试。

### 5.3 Worker 改动

Worker 需要同时处理两类任务：

```
当前逻辑（保持不变）：
  轮询 category=video 且 workflow_id IS NULL 的 generation_records
  → 独立视频任务，完成后直接标记 succeeded/failed

新增逻辑：
  轮询 category=video 且 workflow_id IS NOT NULL 的 generation_records
  → 流水线中的视频步骤，完成后：
    1. 标记 generation_record succeeded/failed
    2. 通知 PipelineEngine.resumeFromAsyncCompletion()
    3. Engine 检查并行任务状态，决定下一步动作
```

**改动方式：** 在 `apps/worker/src/index.ts` 的轮询循环中，除了现有的独立任务处理，增加 pipeline 任务的恢复逻辑。可以通过 PostgreSQL NOTIFY 让 engine 进程响应，也可以让 worker 自己持有 engine 引用直接调用。

推荐方案：**Worker 内持有 PipelineEngine 实例**，因为 worker 已经有了所有的 deps（db、provider、billing）。

```typescript
// apps/worker/src/index.ts（伪代码，展示增量改动）

// 现有逻辑不变
const processor = createTaskProcessor(config)

// 新增：pipeline 引擎
const pipelineEngine = createPipelineEngine({
  createGenerationRecord: ...,
  markSucceeded: ...,
  providerGenerate: ...,
  // 所有 deps 从现有的 db/provider 注入
})

// 轮询循环中新增检查
async function pollPipelineTasks() {
  const records = await pollPipelineVideoTasks()  // workflow_id IS NOT NULL 的视频任务
  for (const record of records) {
    const result = await processor.processTask(record)
    if (result.action === 'completed') {
      // 恢复流水线
      await pipelineEngine.resumeFromAsyncCompletion(record.id, record.outputResult)
    }
  }
}
```

## 6. 费用计算

### 6.1 流水线级别的费用聚合

每一步产生的费用仍然通过现有 `calculateCost()` 计算，写入每条 `generation_record.cost`。

流水线完成后，聚合所有子 record 的费用：

```typescript
// packages/billing/src/workflow-cost.ts（新增）

export function aggregateWorkflowCost(records: GenerationRecord[]): {
  totalCost: number
  steps: { stepName: string; cost: number; records: number }[]
} {
  const steps = new Map<string, { cost: number; records: number }>()

  for (const record of records) {
    if (!record.cost?.totalPrice) continue
    const stepName = record.stepName || 'unknown'
    const existing = steps.get(stepName) || { cost: 0, records: 0 }
    existing.cost += Number(record.cost.totalPrice)
    existing.records += 1
    steps.set(stepName, existing)
  }

  return {
    totalCost: records.reduce((sum, r) => sum + Number(r.cost?.totalPrice || 0), 0),
    steps: Array.from(steps.entries()).map(([stepName, data]) => ({ stepName, ...data })),
  }
}
```

### 6.2 预估费用

在用户点击「开始创作」前，应展示预估费用：

```typescript
// 根据小说长度估算场景数 → 估算视频总时长 → 估算费用
function estimateNovelToVideoCost(
  novelLength: number,
  videoModel: string,
  resolution: string,
): { estimatedScenes: number; estimatedCost: string } {
  const estimatedScenes = Math.max(3, Math.ceil(novelLength / 500))  // 每 500 字一个场景
  const avgDuration = 5
  const model = getModelById(videoModel)
  const pricePerSecond = resolution === '1080P'
    ? (model.pricing.inputPrice1080 || model.pricing.inputPrice)
    : model.pricing.inputPrice
  const textCost = estimatedScenes * 0.01   // LLM 分析成本粗估
  const videoCost = estimatedScenes * avgDuration * pricePerSecond
  return { estimatedScenes, estimatedCost: `¥${(textCost + videoCost).toFixed(2)}` }
}
```

## 8. 风险与注意事项

### 8.1 并行任务的数量控制

小说生视频可能拆出 10-20 个场景，同时发起 20 个视频生成任务会造成：
- DashScope API 限流（并发任务数有上限）
- 费用瞬间飙升

**建议：** 引入并行度上限（如同时最多 3 个视频任务），超出的排队等待。在 `PipelineEngine` 的 `parallel` 动作处理中实现令牌桶或信号量机制。

### 8.2 超时处理

一个完整的流水线可能运行 10-30 分钟（多段视频生成）。需要：
- workflow 级别的超时（如 2 小时）
- 每个步骤的超时继承现有的 4 小时限制
- 超时后标记整个 workflow failed

### 8.3 小说文本长度

qwen-max 的上下文有限（约 32K token），超长小说需要分段处理。`resolveParams` 中需要判断文本长度，超出限制时分批发送。

### 8.4 LLM 输出格式稳定性

场景分析步骤依赖 LLM 输出 JSON。需要：
- 在 prompt 中严格约束 JSON 格式
- 在 `onSucceeded` 中做好 JSON 解析的容错处理
- 解析失败时重试（最多 2 次）

### 8.5 存储空间

多个场景的视频文件会占用较多存储空间。需要：
- 定期清理过期 workflow 的视频文件
- 用户主动删除 workflow 时级联删除文件
- 监控存储用量，设置告警
