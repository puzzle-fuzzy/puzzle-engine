# P1.11: Canvas Pipeline Run 表 — 实施计划

## 背景

当前 Canvas pipeline 没有执行历史记录。9 个 phase 路由都通过 `fireAndForget` 返回 `{ success: true, message }`，没有 runId，没有并发控制，没有开始/结束时间。`canvas_projects.status` 只存当前状态，无法追溯每个 phase 的执行历史。

P1.11 要求：新增 `canvas_pipeline_runs` 表，记录每次 phase 执行；同一 projectId+phase 只允许一个 running；API 返回 `{ accepted: true, runId }`；DB 为权威状态源，SSE 为实时提示。

## 步骤1: Schema + Migration

**新文件** `packages/db/src/schema/canvas-pipeline-runs.ts`:

- pgEnum `canvas_pipeline_phase`: analyze, characters, locations, characterRefs, locationRefs, storyboard, continuity, rebuild, videos (9 个值，retry 不算 pipeline phase)
- pgEnum `canvas_pipeline_run_status`: pending, running, succeeded, failed, cancelled
- Table `canvas_pipeline_runs`:
  - id (uuid PK), projectId (FK), phase (enum), status (enum default 'pending')
  - startedAt, finishedAt (nullable timestamp), errorMessage (text)
  - createdBy (FK accounts), inputSnapshotJson, outputSummaryJson (jsonb)
  - createdAt (timestamp)
  - 无 updatedAt — completed 行不可修改 (append-only)
  - 索引: idx_pipeline_runs_project_phase_status (project_id, phase, status), idx_pipeline_runs_project_created (project_id, created_at)
- 注册到 schema/index.ts
- 添加类型到 types.ts
- 运行 `bun run db:generate` → 0008 migration
- 手动在 migration SQL 中追加 partial unique index: `CREATE UNIQUE INDEX idx_pipeline_runs_one_active_per_phase ON canvas_pipeline_runs (project_id, phase) WHERE status IN ('pending', 'running')`

## 步骤2: Repository 函数

**新文件** `packages/db/src/repositories/canvas-pipeline-runs.repo.ts`:

- `createPipelineRun(values)` — insert, status 默认 'pending'
- `getPipelineRunById(id)` — 单条查询
- `listPipelineRunsByProject(projectId)` — 按项目列出， newest first
- `findActiveRunForPhase(projectId, phase)` — 查找 pending/running 的 run (并发守卫)
- `markPipelineRunRunning(id, inputSnapshot?)` — WHERE status='pending' 才更新
- `markPipelineRunSucceeded(id, outputSummary?)` — WHERE status='running' 才更新
- `markPipelineRunFailed(id, errorMessage)` — WHERE status='running' 才更新
- `markPipelineRunCancelled(id)` — WHERE status IN ('pending','running') 才更新

所有 mark 函数加 status guard (append-only 保护)，status 不匹配时返回 null。

注册到 repositories/index.ts。

## 步骤3: 路由改造 — 返回 `{ accepted, runId }`

修改 `apps/server/src/routes/canvas.ts`:

1. 每个 phase 請求先调用 `findActiveRunForPhase` → 如果有 active run，返回 409 `{ accepted: false, error, existingRunId }`
2. 没有 active run → `createPipelineRun` → 返回 `{ accepted: true, runId }`
3. 替换 `fireAndForget` 为新函数：创建 run 后，传入 `runId` 给 service，service 自己管理状态转换
4. `fireAndForgetWithRun` 只负责 SSE 事件分发和 project-level failed fallback
5. retry endpoint 不创建 run 记录（不是 pipeline phase）

## 步骤4: Service 函数改造 — 接受 runId

修改 `apps/server/src/modules/canvas/service.ts`:

每个 pipeline 函数增加可选 `runId?: string` 参数：
- 入口处：`if (runId) await markPipelineRunRunning(runId, inputSnapshot)`
- 成功处：`if (runId) await markPipelineRunSucceeded(runId, outputSummary)`
- catch 处：`if (runId) await markPipelineRunFailed(runId, errorMessage)`
- `checkContinuity` 和 `rebuildShotPrompts` 需要新加 try/catch（当前没有）

inputSnapshot/outputSummary 是简要摘要（如 `{ charactersCreated: 5 }`），不是完整数据。

## 步骤5: 新增查询端点

- `GET /api/canvas/projects/:projectId/runs` — 列出项目的所有 runs
- `GET /api/canvas/runs/:runId` — 单条 run 状态查询（SSE 断线后 fallback）

## 步骤6: SSE 事件扩展

`pipeline_node_update` 事件新增 `runId` 字段：
- packages/shared/src/canvas.ts — SSEPipelineNodeEvent 加 runId?
