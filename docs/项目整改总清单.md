### 2. `bun run test` 当前失败

失败文件：

- `apps/server/test/generate-routes.test.ts`

失败用例：

- `generate routes > POST /api/generate > referenceFileIds 全部属于当前用户时正常创建记录`

现象：

测试期望 provider 收到：

```ts
{ prompt: '你好' }
```

实际收到：

```ts
{ prompt: '你好', __validatedBrand: true }
```

判断：

- 这是 `ValidatedModelParameters` branded type 引入后的测试断言未同步。
- 不应为了测试通过把业务代码改回弱类型。
- 应修测试 helper/断言，让测试验证业务关键信息：prompt 保留、referenceUrls 正确传递、参数已通过 validated params 路径。

解决办法：

- 在测试中使用 `expect.objectContaining({ prompt: '你好' })` 或抽出 `expectValidatedParams()` helper。
- 不允许继续用 `as any` 构造 branded params。
- 修复后运行：

```bash
bun test --cwd apps/server test/generate-routes.test.ts
bun run test
```

完成定义：

- server 单文件测试通过。
- 根 `bun run test` 通过。
- 此项单独提交 commit，例如 `test: align generate route assertions with validated params`。

### 3. 测试代码仍存在 `as any`

当前命中：

```text
apps/server/test/generate-routes.test.ts
apps/server/test/auth-api-key.test.ts
apps/server/test/auth-guard.test.ts
apps/server/test/upload-routes.test.ts
apps/server/test/generate-routes-retry-cancel.test.ts
apps/server/test/canvas-routes.test.ts
apps/server/test/canvas-routes-extended.test.ts
apps/server/test/upload-routes-delete.test.ts
apps/server/test/openai-gateway.test.ts
```

典型模式：

```ts
validateAndMerge: (...) => ({ ok: true, params: { ...params, __validatedBrand: true } as any })
```

判断：

- 这和项目“测试代码也不允许靠 `any` 糊过去”的原则冲突。
- `ValidatedModelParameters` 是 branded type，测试需要合法的测试构造路径或专用 helper，而不是散落的 `as any`。

解决办法：

- 在 server test helper 中提供一个单一 helper，例如：

```ts
function makeValidatedParams(params: Record<string, unknown>): ValidatedModelParameters
```

- helper 内部使用 `params as ValidatedModelParameters`，并集中注释说明这是测试边界；不要在每个测试文件里写 `as any`。
- 所有 mock `validateAndMerge` 复用该 helper。
- 修复后运行：

```bash
rg -n "\bas any\b|\bany\b|@ts-ignore|@ts-expect-error" apps packages
bun run lint
bun run test
```

完成定义：

- `rg any` 只剩注释、tsconfig 模板注释或确实不可控的第三方声明。
- 测试文件不再出现 `as any`。
- 此项单独提交 commit，例如 `test: remove unsafe any from validation mocks`。

### 5. 路由响应 DTO 语义仍不统一

现状：

- Canvas pipeline phase 和视频重试入口已改为 `{ accepted: true, runId }`。
- 其他 API 仍大量返回 `{ success: true }` / `{ success: false }`。
- 简单同步 mutation 返回 `success` 不一定是 bug，但异步任务、部分成功、冲突、失败不应只靠 success 表达。

解决办法：

- 为常用响应建立明确共享 DTO，例如：
  - `MutationOkResponse`
  - `AcceptedResponse`
  - `ListResponse<T>`
  - `EntityResponse<T>`
  - `ApiErrorResponse`
- 异步入口继续使用 accepted 语义。
- 失败使用 HTTP status + 结构化错误码，不把业务失败都包装成 HTTP 200。

完成定义：

- 新增或重构的路由不再手写零散 response shape。
- 前端 API 层只暴露明确 DTO。

### 6. SSE 认证口径注释与实现不一致

现状：

- 前端 `sseClient.connect()` 已支持刷新后通过 httpOnly cookie 建立连接，并使用 `credentials: 'include'`。
- 服务端 `createAuthPlugin()` 也支持 cookie、API Key Bearer、JWT Bearer。
- 但 `apps/server/src/routes/sse.ts` 注释和 OpenAPI description 仍写“仅支持 Bearer header 认证”。

风险：

- 后续维护者可能误以为刷新后必须恢复内存 token，重新引入 localStorage 或 query token。

解决办法：

- 更新 SSE 路由注释和 description：
  - 浏览器端优先 httpOnly cookie。
  - 编程式客户端可使用 `Authorization: Bearer <jwt>`。
  - query token 不支持。
- 补一条服务端 SSE 认证测试：cookie JWT 可访问 `/api/sse`，无认证返回 401。

完成定义：

- 注释、OpenAPI 描述、前端实现、服务端认证实现一致。
- 此项单独提交 commit。

### 7. dedupeKey 仍使用 `JSON.stringify(parameters)`

位置：

- `apps/server/src/routes/generate.ts`
- `apps/server/src/routes/openai-gateway.ts`

现状：

```ts
const dedupeKey = `${userId}:${model}:${JSON.stringify(parameters)}`
```

问题：

- key 顺序敏感。
- prompt 或参数较长时 key 过长。
- OpenAI gateway 也用了同类策略。
- 当前 schema 中 `dedupeKey` 已是 `text`，但语义仍不够稳定。

解决办法：

- 使用 canonical stringify + SHA-256。
- dedupeKey 建议为 `sha256(userId + model + canonicalParams)`。
- 普通 generate 和 OpenAI gateway 共用同一个 helper。
- 增加测试：不同 key 顺序的等价参数命中同一 dedupeKey。

## P2：质量与产品化改进

### 8. fire-and-forget 测试仍会产生后台错误日志

现象：

- `bun run test` 中多个 Canvas route 测试虽然通过，但后台任务在 route 返回后报“项目不存在”“镜头不存在”“没有失败的镜头可以重试”等错误日志。

判断：

- 当前测试关注“立即返回 accepted”，但没有很好地隔离 fire-and-forget 后台任务。
- 日志不是失败，但会污染测试输出，掩盖真实错误。

解决办法：

- route 测试中 mock phase service，使后台任务可控完成。
- 或为 fire-and-forget helper 注入 test scheduler，在测试中同步 flush。
- 保留至少一组测试覆盖后台失败会写 pipeline run failed。
