## P0：必须修复

### 1. ProviderResult 还不是 discriminated union

问题：

- [ ] `packages/provider/src/types.ts` 中 `ProviderResult` 仍是 `success: boolean` + `providerTaskId?` + `output?: Record<string, unknown>`。
- [ ] 这不是原计划中的 `TextOutput | ImageOutput | VideoTaskOutput | VideoResultOutput` 可辨识联合。
- [ ] server 仍需要靠 `result.providerTaskId`、`result.output` 等可选字段判断分支，编译器不能保证 exhaustiveness。

解决办法：

- 改为类似：

```ts
type ProviderResult =
  | { type: 'text'; model: string; output: TextOutputResult; usage?: ProviderUsage }
  | { type: 'image'; model: string; output: ImageOutputResult; usage?: ProviderUsage }
  | { type: 'video_task'; model: string; taskId: string }
  | { type: 'failed'; model?: string; error: string }
```

- `DashScopeClient` 各方法返回明确 variant。
- generation service / worker / Canvas video 入口按 `type` switch，并加 exhaustive check。
- 补 provider 和 server route 测试。

完成程度：50%。已有 output parser，但 provider 边界本身仍是宽类型。

### 2. Canvas 视频生成入口未复用模型参数校验

问题：

- [ ] `apps/server/src/modules/canvas/videos.ts` 直接构造 `videoParams` 并调用 `submitVideoTaskWithFallback()`。
- [ ] `validateModelParameters()` 只在 `/api/generate` 和 OpenAI 网关中出现，未在 Canvas `generateVideos()` / `retryShotVideo()` / `retryFailedShots()` 中调用。
- [ ] 原文档曾写“Canvas 入口未来集成”，但该项被标记完成，验收不通过。

解决办法：

- 在 Canvas 选择最终视频模型后，调用 `getModelById()`。
- 对 `videoParams` 调用 `validateModelParameters()`，通过后再 `mergeWithDefaults()`。
- 校验失败时将对应 shot 标记为 `failed`，错误写入 `errorMessage` 并通过 SSE 通知。
- 增加测试：非法 duration / resolution / 未知参数不会提交 provider task。

完成程度：70%。通用校验器已存在，Canvas 集成缺失。

### 3. CanvasLayoutDto 仍是裸 `Record<string, unknown>`

问题：

- [ ] `packages/db/src/domain-types.ts` 中 `CanvasLayoutDto = Record<string, unknown>`。
- [ ] `apps/server/src/routes/canvas.ts` 的 layout body 使用 `t.Record(t.String(), t.Any())`。
- [ ] 这与“Canvas JSON 字段建立明确类型”的完成声明不一致。

解决办法：

- 定义最小 React Flow 布局 DTO，例如 `nodes`、`edges`、`viewport`。
- 写 `parseCanvasLayout()`，只在边界接受 unknown，落库前转成明确 DTO。
- 路由 schema 替换 `t.Any()`。
- 增加非法 layout 请求测试。

完成程度：40%。目前只是透传，不是强类型解析。

## P1：上线前应修复

### 4. 审计日志在普通 server 测试中访问真实 DB

问题：

- [ ] `bun run test` 虽然通过，但 server 测试输出多次 `relation "audit_logs" does not exist`。
- [ ] 说明部分 route/service 测试没有隔离审计日志副作用，测试通过依赖 audit 写入失败被吞掉。
- [ ] 这会降低审计功能真实可用性的信心。

解决办法：

- 将 audit service 做成可注入依赖，测试中使用 noop/mock。
- 或在 route factory config 中提供 `audit?: AuditService`。
- 针对审计写入单独补集成测试，使用完整 migrated DB。

完成程度：65%。功能入口存在，但测试隔离不足。

### 5. 通用 workflow 只完成 DB/repository，未形成可用引擎

问题：

- [ ] 当前只看到 `workflows`、`workflow_steps` schema 和 repository。
- [ ] 没有 worker/orchestrator、route、状态推进服务、步骤执行器或端到端测试。
- [ ] 原清单中“通用流水线引擎完成”表述过度。

解决办法：

- 如果近期不做产品入口，将该项降级为“数据模型预研完成”。
- 如果要继续做，补 `workflow-runner`、锁心跳续约、步骤执行协议、恢复扫描和失败重试测试。

完成程度：25%。数据层完成，应用层未完成。

### 6. 仍有较多 `Record<string, unknown>` 跨模块传播

问题：

- [ ] provider params、billing params、generation inputParams、workflow input/output、audit detail、notify payload、workspace parameters 均大量使用 `Record<string, unknown>`。
- [ ] 其中 DB JSONB 和外部 provider 边界可以保留，但部分已经进入 service/store 层长期传播。

解决办法：

- 区分“存储 JSONB 类型”和“业务 DTO 类型”。
- 对 generation parameters 建立 `ValidatedModelParameters` branded type，只有校验器能构造。
- workflow input/output 按 workflow type 定义 schema，不在服务层直接传裸 record。

完成程度：70%。边界解析有所改善，但还未达到原清单的严格目标。

### 7. 路由返回仍大量使用 `{ success: true }`

问题：

- [ ] Canvas pipeline phase 已改为 `{ accepted: true, runId }`，但其他 API 仍大量返回 `{ success: true }`。
- [ ] 这不一定是 bug，但与“DTO 语义清晰、不用 success 包装所有状态”的方向不完全一致。

解决办法：

- 保留 mutation 的简单成功响应可以接受，但需要显式 DTO 类型命名。
- 对异步任务、部分成功、冲突、失败等状态避免只用 `success` 表达。

完成程度：75%。关键异步入口已改善，普通 API 仍需 DTO 收敛。
