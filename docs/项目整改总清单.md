## P1：上线前应修复

### 1. 仍有较多 `Record<string, unknown>` 跨模块传播

问题：

- [ ] provider params、billing params、generation inputParams、workflow input/output、audit detail、notify payload、workspace parameters 均大量使用 `Record<string, unknown>`。
- [ ] 其中 DB JSONB 和外部 provider 边界可以保留，但部分已经进入 service/store 层长期传播。

解决办法：

- 区分“存储 JSONB 类型”和“业务 DTO 类型”。
- 对 generation parameters 建立 `ValidatedModelParameters` branded type，只允许校验器构造。
- workflow input/output 按 workflow type 定义 schema，不在服务层直接传递 record。

完成程度：25%。ProviderResult、Canvas 视频参数、Canvas layout 已收紧，但整体 DTO 边界仍未系统化。

### 2. 路由返回仍大量使用 `{ success: true }`

问题：

- [ ] Canvas pipeline phase 和视频重试入口已改为 `{ accepted: true, runId? }`，但其他 API 仍大量返回 `{ success: true }`。
- [ ] 这不一定是 bug，但与“DTO 语义清晰、不用 success 包装所有状态”的方向不完全一致。

解决办法：

- 简单 mutation 的成功响应可以保留，但需要显式 DTO 类型命名。
- 对异步任务、部分成功、冲突、失败等状态避免只用 `success` 表达。

完成程度：20%。关键异步入口和视频重试入口已改善，普通 API 仍需 DTO 收敛。
