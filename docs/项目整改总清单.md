# 项目整改总清单

整理时间：2026-06-11

本文是项目后续整改的唯一入口文档。后续给 Claude 或其他 AI 助手分派任务时，只读取本文；`docs/bailian` 仅作为阿里云百炼资料库，不参与项目整改口径。

## 总原则

1. 项目尚未上线，不保留历史兼容逻辑，不迁就旧接口、旧状态、旧鉴权路径或旧数据形态。
2. TypeScript 类型必须完整。目标是业务源码零 `any`，尽量减少裸 `unknown`；`unknown` 只允许出现在外部输入边界，并且必须在边界层立即解析成领域类型。
3. 测试必须验证真实风险，不能为了让测试过而放宽类型、塞 `as any` 或跳过断言。
4. 架构设计优先于临时补丁。跨 server/client/worker/db/provider 的能力必须有清晰边界和状态机。
5. 文件保持单一职责。路由、service、mapper、validator、repository、client API、UI 组件不要互相吞职责。
6. 核心功能必须有必要注释。注释解释业务约束、状态机、外部 API 限制和恢复逻辑，不解释显而易见的语法。
7. 完成一项独立整改后就提交对应代码。不要把类型修复、状态机、SSE、计费、UI 等多类改动混进一个大提交。

## 项目现状

### 背景判断

- 当前工程底座优于参考项目：Bun monorepo、server/client/worker 三进程、Eden Treaty、Drizzle、PostgreSQL、SSE、Pino、测试基础都应保留。
- 不照搬参考项目的降级做法：不改回手写 fetch client、不把视频轮询塞回 server、不改 SQLite、不放弃认证、不用 console 日志、不放弃测试。
- 参考项目中值得吸收的是领域深度、工作流状态机、账务闭环、可观测性和部署文档，不是它的工程弱点。
- 通用 pipeline 思路可以作为未来方向，但必须重写成强类型设计，不使用 `any` 或长期漂浮的 `Record<string, unknown>`。

### 已完成基础

已完成内容进入“复核但不重复实现”状态：

- Canvas 项目、角色、场景、镜头的基础 CRUD 字段补全。
- 单条角色/场景/镜头删除、上传文件删除。
- 生成任务 retry/cancel、Canvas 单镜头重试。
- 前端 toast 错误反馈和 ErrorBoundary。
- per-user 计费统计。
- 整数分计费基础字段和展示。
- location refs 状态、locked 角色/场景保留。
- 复合索引、Worker poll limit、Worker 优雅退出。
- Swagger/OpenAPI UI 基础接入。
- PromptEditor、ReferenceUploadZone。
- Canvas mapper 基础实现。
- SSE 客户端已迁移到 `@microsoft/fetch-event-source`。

### 当前取舍

- 未上线项目可以直接调整 schema、接口、状态机和鉴权方式，不写历史兼容分支。
- SSE 只保留 `Authorization: Bearer <jwt>`，移除 query token。
- 当前技术路线是 Bun，不引入 Node 双运行时。
- 核心账务以 `totalPriceCents` 为准，浮点金额只允许作为展示派生值。
- OpenAI 格式网关属于未来产品决策，不进入当前架构整改 P0/P1。

## 编码硬规则

### TypeScript

1. 业务源码禁止新增 `any`、`as any`、`@ts-ignore`、`@ts-expect-error`。
2. `unknown` 只允许用于外部输入边界：JSON、SSE、provider response、Eden raw response、DB JSONB、catch error。
3. `unknown` 必须在边界函数内解析成明确 DTO 或 discriminated union，不允许跨 service/store/component 继续传播。
4. 类型断言只能用于已经被运行时守卫证明的地方，并且优先写 parser/type guard。
5. 所有核心 union 必须做 exhaustive check，新增状态时让编译器提示漏处理。
6. 测试代码也要保持类型正确，不能用 `as any` 掩盖 mock 类型问题。

### API 与 DTO

1. 路由层返回 DTO，不直接返回 DB row。
2. DB row -> DTO 放在 mapper。
3. 请求体和 JSONB 字段解析放在 validator/parser。
4. client API 只暴露解析后的 DTO，不把 Eden 原始 `{ data, error }` 泄漏到页面。
5. 业务错误使用正确 HTTP status，不用 HTTP 200 包装所有失败。

### 文件职责

1. `routes`：HTTP schema、认证、调用 service。
2. `service`：业务流程和状态机。
3. `repository`：数据库读写，不写业务判断。
4. `mapper`：DB row 到 DTO。
5. `validator/parser`：外部输入和 JSON 字段解析。
6. `store`：前端跨页面状态，不直接塞复杂请求流程。
7. React 组件只负责渲染和用户交互，不承载跨模块业务编排。

### 测试

1. 每个修复至少覆盖一个能失败的测试或说明已有测试覆盖。
2. Mock 使用明确泛型和 test factory。
3. 非法输入测试通过 parser/validator 验证失败，不靠 `any` 构造。
4. DB schema 变化必须带 migration，并补 DB 测试或说明验证方式。
5. SSE、状态机、计费、权限、文件上传属于高风险区域，改动必须有专项测试。

### 注释

1. 核心状态机、账务不变量、并发锁、外部 API 限制、恢复策略必须写注释。
2. 注释解释“为什么”和“边界条件”，不解释显而易见的语法。
3. 临时限制必须写明删除条件，不能留下模糊 TODO。

### 3. 约束 `unknown` 的使用边界

允许场景：

- `JSON.parse()`、SSE payload、provider response、Eden `response.data`、数据库 JSONB 字段读取。
- catch error。
- 测试中表示外部输入。

禁止场景：

- service 层核心业务对象长期使用 `Record<string, unknown>`。
- client API 返回 `{ data: unknown }` 后继续传给页面。
- pipeline context、model params、canvas layout 在多个模块之间以裸 `unknown` 传播。

待办：

- [ ] 为 Canvas JSON 字段建立明确类型：`CanvasLayoutDto`、`CharacterProfileJson`、`LocationProfileJson`、`ShotCameraJson`、`ShotTimelineJson`。
- [ ] 为 Model parameters 建立按模型配置推导或运行时校验后的 `ValidatedModelParameters`。
- [ ] 为 SSE event payload 使用 `SSEEventMap` + parser，事件分发前完成解析。
- [ ] 为 Eden API 响应建立 DTO parser，`unwrapEden<T>` 不能只做 `response.data as T`。

## 提交规则

- 每完成一个独立待办，就运行对应验证命令并创建一次 git commit。
- commit 范围只包含该待办相关代码、测试、migration 和文档更新。
- 不把无关格式化、临时调试、其他待办的半成品混入同一个 commit。
- commit message 使用清晰动词和范围，例如 `fix: remove unsafe any from sse client`、`test: type auth route mocks`、`refactor: add generation dto mappers`。
- 如果某项待办必须拆成多步，按可验证边界拆成多个 commit。

## 完成定义

一次整改 PR 或 Claude 任务必须满足：

- [ ] 没有新增 `any`、`as any`、`@ts-ignore`。
- [ ] 所有新增 `unknown` 都在同一边界函数内被解析成具体类型。
- [ ] 类型、lint、相关测试通过。
- [ ] 涉及 schema 的修改带 migration。
- [ ] 涉及核心状态机、账务、外部 API 或并发的逻辑有必要注释。
- [ ] 文档中对应待办被标记完成或补充新的后续项。
- [ ] 每个已完成的独立待办都有对应 git commit。
