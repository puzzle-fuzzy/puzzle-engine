## 一、Canvas CRUD 补全（高优先级）

### 1.1 项目标题/故事文本不可编辑 ✅ 已完成

**问题：** 用户创建项目后无法修改 `title` 和 `storyText`，写错只能删项目重建。

**位置：**
- 路由层：`apps/server/src/routes/canvas.ts` — 没有 `PUT/PATCH /projects/:projectId` 通用更新端点
- Service 层：`apps/server/src/modules/canvas/service.ts` — 没有 `updateProjectProperties()` 函数
- DB 层：`packages/db/src/repositories/canvas-projects.repo.ts:32-42` — `updateCanvasProject()` 的类型约束只允许 `title | status | analysisJson | modelPreferencesJson | canvasLayout`，**不包含 `storyText`**

**解决方案：**
1. `packages/db/src/repositories/canvas-projects.repo.ts` — 扩展 `updateCanvasProject` 的 `values` 类型，加入 `storyText`
2. `apps/server/src/modules/canvas/service.ts` — 新增 `updateProjectProperties(projectId, { title?, storyText? })`
3. `apps/server/src/routes/canvas.ts` — 新增路由：
   ```
   PATCH /api/canvas/projects/:projectId
   body: { title?: string, storyText?: string }
   ```
4. 前端 `NodeDetailPanel.tsx` 或 `CanvasEditor.tsx` — 给 Analysis 节点加编辑按钮，可改 title/storyText

### 1.2 角色属性部分不可修改 ✅ 已完成

**问题：** 角色 PATCH 端点只接受 `identityPrompt`、`negativePrompt`、`locked`，不允许修改 `name`、`role`、`description`、`referenceImageUrl`。LLM 生成的名字/角色定位不满意时无法手动修正。

**位置：**
- 路由：`apps/server/src/routes/canvas.ts:148-152` — `PATCH /characters/:characterId` body 约束为 `{ identityPrompt?, negativePrompt?, locked? }`
- Service：`apps/server/src/modules/canvas/service.ts` — `updateCharacterData()` 只传这三个字段
- DB 层：`packages/db/src/repositories/canvas-characters.repo.ts:34-44` — `updateCanvasCharacter()` 实际支持 `name | role | description | identityPrompt | negativePrompt | profileJson | referenceImageUrl | turnaroundSheetUrl | locked`，但路由层没暴露

**解决方案：**
1. `apps/server/src/routes/canvas.ts:148` — 扩展 PATCH body schema，加入 `name?`、`role?`、`description?`、`referenceImageUrl?`
2. `apps/server/src/modules/canvas/service.ts` — `updateCharacterData()` 透传所有字段
3. 前端 `NodeDetailPanel.tsx` — Character 详情面板加 name/role/description 输入框

## 二、生成任务生命周期（高优先级）

### 2.1 无法重试失败任务 ✅ 已完成

**问题：** 失败的 generation_record 永远停在 `failed`，没有 retry 端点或自动重试逻辑。用户必须从零创建新请求。

**位置：**
- 路路由缺失：`apps/server/src/routes/generate.ts` — 无 `POST /api/records/:id/retry`
- DB 层缺失：`packages/db/src/repositories/generation-records.repo.ts` — 无 `resetGenerationToPending(id)`
- Worker：`apps/worker/src/task-processor.ts` — 只处理 pending/processing 状态

**解决方案：**
1. `packages/db/src/repositories/generation-records.repo.ts` — 新增 `resetGenerationToPending(id)` 将 status 改回 `pending`，清空 errorMessage
2. `apps/server/src/routes/generate.ts` — 新增 `POST /api/records/:id/retry`，校验归属 + 状态为 failed，调用 reset + 重新提交 DashScope 任务
3. Canvas shot 重试：新增 `POST /api/canvas/shots/:shotId/retry`，只重跑单个失败镜头（当前 `generateVideos` 会全量重跑所有镜头）
4. 前端 Workspace + Assets 页面：failed 记录旁加"重试"按钮

---

### 2.2 无法取消进行中的任务 ✅ 已完成

**问题：** 没有 cancel 端点，Worker 会一直轮询直到 4 小时超时。用户无法主动终止正在跑的视频生成。

**位置：**
- 路路由缺失：`apps/server/src/routes/generate.ts` — 无 `POST /api/records/:id/cancel`
- Provider 缺失：`packages/provider/src/dashscope-client.ts` — 无 `cancelTask(taskId)` 方法
- Worker：`apps/worker/src/task-processor.ts` — 没有 cancel 检查逻辑

**解决方案：**
1. `packages/provider/src/dashscope-client.ts` — 新增 `cancelTask(taskId)` 调用 DashScope 取消 API
2. `packages/db/src/repositories/generation-records.repo.ts` — 新增 `cancelGenerationRecord(id)` 设置 status = `failed`, errorMessage = '用户取消'
3. `apps/server/src/routes/generate.ts` — 新增 `POST /api/records/:id/cancel`
4. Canvas: `POST /api/canvas/shots/:shotId/cancel`
5. 前端：processing 记录旁加"取消"按钮

---

### 2.3 Canvas 视频只能全量重新生成 ✅ 已完成

**问题：** 如果 9 个镜头中 1 个失败，重跑 `generate-videos` 会覆盖所有镜头（包括已成功的），因为 `generateStoryboard` 重新运行时会 `deleteCanvasShotsByProject`。

**位置：**
- `apps/server/src/modules/canvas/service.ts:565-669` — `generateVideos` 逐个提交但不跳过已有视频的 shot
- `apps/server/src/modules/canvas/service.ts:195,250` — `generateCharacters`、`generateLocations` 重新运行时会 `deleteCanvasShotsByProject`

**解决方案：**
1. `generateVideos` 中加判断：如果 shot 已有 `videoUrl` 且 `status === 'completed'`，跳过该镜头
2. 新增 `POST /api/canvas/shots/:shotId/retry` 端点（见 2.1），支持只重跑单个失败镜头
3. `generateStoryboard` 重新运行时：保留 `status === 'completed'` 的 shots，只删除 draft/ready/failed 状态的
4. Service 层新增 `retryShotVideo(shotId, config)` 函数，只对单个 shot 提交视频任务

## 三、前端错误反馈与体验（中优先级）

### 3.1 多处 catch 静默吞掉错误

**问题：** 前端至少 7 个 catch 块是空的（`catch {}` 或 `catch(() => {})`），用户操作失败时没有任何反馈，以为操作成功了。

**位置：**
- `apps/client/src/pages/Workspace.tsx:141` — `catch(() => {})`：加载 canvas 项目列表失败
- `apps/client/src/pages/Workspace.tsx:162` — `catch {}`：加载生成记录失败
- `apps/client/src/pages/Workspace.tsx:256` — `catch {}`：删除记录失败
- `apps/client/src/pages/Workspace.tsx:271` — `catch {}`：另一处操作失败
- `apps/client/src/pages/Workspace.tsx:316` — `catch {}`：操作失败
- `apps/client/src/pages/Assets.tsx:36` — `catch {}`：加载资产列表失败
- `apps/client/src/pages/Billing.tsx:28` — `catch(() => {})`：加载计费统计失败
- `apps/client/src/components/canvas/PipelineController.tsx:98` — `catch(() => {})`：pipeline 步骤请求失败

3. 对有 `err` 参数但没处理的 catch（如 Canvas.tsx:53,72,89），也加 toast
4. AuthProvider 的 catch 至少需在 401 时跳转登录页

## 四、计费功能补全（中优先级）

### 4.1 计费统计端点只返回全局聚合

**问题：** `GET /api/billing/statistics` 返回全系统的费用统计，不是 per-user 的。用户看不到自己的花费明细。且该端点无认证（见 TODO.md 安全部分，但此处只谈功能）。

**位置：**
- `apps/server/src/routes/billing.ts:6` — `getCostRecords()` 无 accountId 过滤
- `packages/db/src/repositories/generation-records.repo.ts:131-144` — `getCostRecords()` 拉全部 cost 非空记录

**解决方案：**
1. `packages/db/src/repositories/generation-records.repo.ts` — `getCostRecords()` 新增可选参数 `{ accountId, startDate, endDate }`
2. `apps/server/src/routes/billing.ts` — 改为 `GET /api/billing/statistics?accountId=me`（"me" 解析为当前 userId）
3. 前端 `Billing.tsx` 默认只看自己的数据
4. 加 auth plugin 到 billing 路由

---

### 4.2 计费浮点数运算

**问题：** 所有金额计算使用 `roundTo4()` 浮点数（4 位小数），未转为整数分。浮点运算可能导致金额偏差（如 0.1 + 0.2 !== 0.3）。

**位置：**
- `packages/billing/src/calculate.ts:85-87` — `roundTo4()` 浮点四舍五入
- `packages/billing/src/calculate.ts:27-29` — token 计费 `(tokens / 1_000_000) * price` 浮点乘法
- `packages/billing/src/statistics.ts` — aggregateStatistics 也用浮点

**解决方案：**
1. 所有价格配置改为**整数分**（CNY cents）：`inputPrice: 0.002` → `inputPriceCents: 200`
2. `calculateCost()` 内部运算全部用整数：`(tokens * inputPriceCents) / 1_000_000`
3. 最终返回值同时包含 `totalPriceCents` 和 `totalPrice`（cents / 100 转元，展示用）
4. `packages/provider/src/model-configs.ts` — 所有 pricing 字段改为 cents
5. 前端展示层：`totalPriceCents / 100` → 显示金额

---

### 4.3 getCostRecords 全表扫描 + JS 过滤

**问题：** `getCostRecords()` 拉所有 cost 非空的记录到内存，再用 JS 过滤 totalPrice 是否为数字。无分页、无日期范围。

**位置：**
- `packages/db/src/repositories/generation-records.repo.ts:131-144`

**解决方案：**
1. SQL 层直接聚合：`SELECT category, model, SUM((cost->>'totalPrice')::numeric) as total, ...` 而不是拉全表
2. 支持日期范围：`WHERE created_at BETWEEN $1 AND $2`
3. 支持 accountId 过滤
4. 如果不想依赖 JSONB 内部查询，可以新增 `total_price_cents` 整数列（见 4.2），直接 SQL SUM

### 5.2 analyzeProject 重新分析时不尊重 locked 角色和场景

**问题：** `analyzeProject` 在重新分析时会 `deleteCanvasCharactersByProject(projectId)` 和 `deleteCanvasLocationsByProject(projectId)`，不传 `excludeLocked`，导致用户锁定的角色/场景也被删除。

**位置：**

   ```typescript
   await deleteCanvasCharactersByProject(projectId, { excludeLocked: true })
   await deleteCanvasLocationsByProject(projectId, { excludeLocked: true })
   ```
2. DB 层已支持此参数（`canvas-characters.repo.ts:27` 和 `canvas-locations.repo.ts:27`）

### 5.3 Fallback 逻辑硬编码在 service 层

**问题：** `model-configs.ts` 已有声明式 `fallbackModel` 字段，但 `service.ts` 仍然硬编码判断 `referenceUrls.length > 0 ? 'happyhorse-1.0-r2v' : 'happyhorse-1.0-t2v'`。

**位置：**
- `apps/server/src/modules/canvas/service.ts:596` — 硬编码模型选择逻辑
- `apps/server/src/modules/canvas/service.ts:598` — 读 `fallbackModel` 但只在失败时使用
- `packages/provider/src/model-configs.ts` — `happyhorse-1.0-r2v` 和 `wan2.7-r2v` 已有 `fallbackModel` 字段

**解决方案：**
1. Service 层改为从 `modelPreferencesJson` 读取用户选择的视频模型（当前硬编码 happyhorse）
2. 根据 `referenceUrls` 判断用 r2v 还是 t2v 版本的选中模型
3. Fallback 提取到 provider 层：`packages/provider/src/dashscope-client.ts` 新增 `submitWithFallback(model, params, referenceUrls)` 方法，失败自动尝试 `fallbackModel`
4. `generate.ts` 的通用生成路由也需要同样的 fallback 支持

## 六、性能优化（中优先级）

### 6.1 listProjects N+1 查询

**问题：** 每个项目调用 `getCanvasProjectDetail()` 产生 4 条子查询（characters, locations, shots, continuity）。用户有 N 个项目 = 1 + 4N 条 SQL。

**位置：**
- `apps/server/src/modules/canvas/service.ts:120-126`

**解决方案：**
1. 新增 `batchGetCanvasProjectDetails(accountId)` 函数：一条 SQL JOIN 查询拉所有项目 + 关联数据
2. 或分批查询：先拉所有项目 ID → 一条 SQL 拉所有 characters → 一条拉 locations → 一条拉 shots → JS 端按 projectId 分组
3. `packages/db/src/repositories/canvas-projects.repo.ts` — 新增批量查询函数

---

### 6.2 reconcileProjectShots 每次读项目详情都执行

**问题：** `getProjectDetail()` 每次调用都先跑 `reconcileProjectShots()`，即使没有需要修复的 stale shot。这增加了一次 `listCanvasShotsByProject` + `getGenerationRecordsByTaskIds` 查询。

**位置：**
- `apps/server/src/modules/canvas/service.ts:68-110`

**解决方案：**
1. `reconcileProjectShots` 改为独立端点 `POST /api/canvas/projects/:id/reconcile`，用户或前端在需要时主动调用
2. 或在 `getProjectDetail` 中用缓存标记：项目 status 不是 `generating` 时跳过 reconcile
3. 最简方案：只在 Worker 处理完 canvas shot 任务后自动调用 reconcile，而不是每次读取时执行

---

### 6.3 Worker pollPendingVideoTasks 无限制

**问题：** 每 5 秒拉全部 pending/processing 视频记录，量大时查询变慢。

**位置：**
- `packages/db/src/repositories/generation-records.repo.ts:108-118`

**解决方案：**
1. 添加 `LIMIT 50` 防止单次拉取过多记录
2. 添加复合索引 `(status, category)` 加速查询（见 6.4）

---

### 6.4 缺少复合索引

**问题：** 高频查询列没有显式索引，依赖 PostgreSQL 主键/唯一索引。

**需要添加的索引：**

| `canvas_locations` | `(project_id)` | 项目详情查询 |
| `canvas_shots` | `(project_id, shot_index)` | 项目详情查询 |
| `uploaded_files` | `(account_id)` | 用户文件列表 |

**位置：**
- `packages/db/src/schema/*.ts` 各表定义文件

**解决方案：**
1. 在各 schema 文件中使用 Drizzle 的 `.index()` 方法声明索引
2. 运行 `bunx drizzle-kit generate` 生成迁移
3. `bunx drizzle-kit push` 应用到数据库

## 七、Worker 可靠性（中优先级）

### 7.1 Worker 优雅退出不等待当前任务完成

**问题：** 收到 SIGINT/SIGTERM 时只设 `running = false`，不等待当前 `processTask` 完成。如果任务正在下载视频文件，可能导致 shot 状态停在 `generating` 但视频 URL 永远不会写入。

**位置：**
- `apps/worker/src/index.ts:13-18` — 信号处理只设 flag，不等待
- `apps/worker/src/index.ts:33` — `processTask(record)` 无中断机制

**解决方案：**
1. 将当前正在处理的任务包装为 Promise，退出信号时 `await` 该 Promise
2. 或设置超时：最多等待 30 秒完成当前任务，超时后强制退出
3. 退出前将正在处理的 shot 状态标记回 `failed`（防止永远停在 generating）

### 7.2 audio 类别在 DB enum 中但零实现

**问题：** `generation_category` enum 包含 `audio`，但没有音频模型配置、没有 billing 计算逻辑、没有 Worker 处理。如果代码不小心设 category = `audio`，calculateCost 返回 `{ unit: 'unknown', totalPrice: 0 }`。

**位置：**
- `packages/db/src/schema/generation-records.ts:10` — `generationCategoryEnum` 包含 `'audio'`
- `packages/billing/src/calculate.ts:73-74` — `default` 返回 `{ unit: 'unknown', totalPrice: 0 }`

**解决方案：**
1. 短期：从 `generationCategoryEnum` 移除 `audio`（没有对应功能就不该有 enum 值）
2. 长期（需要音频功能时）：加回 `audio` + 对应的 model config + billing 逻辑 + Worker 处理

**问题：** 没有 `retry_count` 字段来跟踪重试次数，没有 `dedupe_key` 防同参数重复提交。

**解决方案：**

2. `dedupe_key` = `model + hash(params)` 组合，POST /generate 时先查 dedupe_key 是否已存在
3. retry 时 `retry_count++`（见 2.1）
